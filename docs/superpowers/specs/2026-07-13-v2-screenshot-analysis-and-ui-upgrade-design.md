# 안심스캔 v2 — 스크린샷 분석 + UI 업그레이드 설계 문서

- Date: 2026-07-13
- Status: Draft (approved for planning)
- 선행 문서: `2026-07-07-korean-scam-detector-design.md` (v1 설계 — 아키텍처, 보안/개인정보 원칙은 이 문서에서 그대로 승계한다)

## 1. 개요 (Overview)

v1은 문자(SMS)/이메일 **텍스트 붙여넣기**만 지원했다. 실제 사용 패턴에서 사람들은 의심스러운 메시지를 타이핑하거나 복사하기보다 **스크린샷을 찍는다** — v2의 핵심은 이 흐름을 직접 지원하는 것이다.

**v2 범위 (두 가지 워크스트림):**

1. **스크린샷 분석**: 메시징 앱 종류를 불문하고(SMS, 카카오톡, 이메일 클라이언트 등) 스크린샷 1-5장을 업로드하면 멀티모달 LLM이 내용을 판독하고 기존과 동일한 위험도 판정을 반환한다
2. **UI 업그레이드**: 모바일 우선 재작업 + 결과 화면 강화(원문 내 의심 문구 하이라이트) + 전반적 비주얼 폴리시

**v2 비범위**: 전체 플로우 재설계(랜딩 페이지 등), URL 목적지 방문 분석, 로그인/이력, 유료 모델 전환. 예산 방침은 v1과 동일하게 **무료 티어 전용**을 유지한다.

## 2. 목표 / 비목표 (Goals / Non-goals)

**목표**:
- 스크린샷 1-5장 업로드 → 단일 LLM 호출로 판독 + 분석 (무료 티어 쿼터를 아끼는 핵심 제약)
- 모델이 판독한 내용(transcript)을 사용자에게 그대로 보여줘 투명성 확보 — 무엇을 읽고 판단했는지 확인 가능
- 모든 redFlag가 메시지 원문의 **실제 인용구(evidence)** 에 근거하도록 스키마 수준에서 강제 — v1에서 프롬프트로만 막았던 "근거 없는 신호 날조"를 구조적으로 차단하고, 동시에 UI 하이라이트의 데이터 소스가 됨
- 폰에서 한 손으로 쓸 수 있는 모바일 우선 UI
- v1의 보안/개인정보 원칙(무저장, 무로깅, 주입 방어, 게이트 순서)을 이미지 입력에도 동일하게 적용

**비목표 (v2)**:
- OCR 전용 단계 분리 (transcribe는 분석 호출 안에서 함께 수행)
- 이미지의 서버 측 저장, 캐싱, 썸네일 생성 — 이미지는 요청 스코프 내에서만 존재
- 스크린샷 진위 판별(합성/조작 탐지)
- 다국어 지원

## 3. 아키텍처 개요

v1 아키텍처(단일 Next.js 앱, Route Handler, DB 없음, Vercel 배포)를 그대로 유지하고 입력 모드 하나를 추가한다:

```
Browser
  ├─ 입력 모드 토글: 텍스트(SMS/이메일) | 스크린샷
  ├─ [이미지 모드] 파일 선택/드래그앤드롭/붙여넣기 (1-5장, accept="image/*" → 모바일에서 카메라/사진첩)
  ├─ [이미지 모드] 클라이언트 측 다운스케일 (canvas, 장변 ≤1280px, JPEG ~0.8)
  └─ POST /api/analyze  (base64 이미지를 기존 JSON 바디에 포함)
        │
Route Handler (게이트 순서 v1과 동일: Turnstile → IP rate limit → 전역 쿼터 → LLM)
  └─ analyzeMessage()
       ├─ [텍스트] Gemini flash-lite → (429 시) Groq gpt-oss-20b  ← v1 그대로
       └─ [이미지] Gemini flash-lite 멀티모달 → (429 시) Groq Llama 4 Scout
            └─ { verdict, riskScore, redFlags[{flag, evidence}], explanation,
                 recommendedAction, extractedText } 반환
```

## 4. 이미지 입력 설계

### 4.1 클라이언트 측 전처리 (필수, 최적화 아님)

원본 폰 스크린샷은 장당 2-5MB로, 5장이면 Vercel의 요청 바디 한도(4.5MB)를 초과한다. 업로드 전 canvas로 **장변 ≤1280px, JPEG 품질 ~0.8**로 다운스케일한다:

- 장당 ~400KB 이하 → 5장 × 1.33(base64 오버헤드) ≈ 2.7MB로 한도 내 안착
- Gemini의 이미지 토큰 비용(768×768 타일당 258토큰)도 절반 수준으로 감소
- 1280px이면 메시징 앱 스크린샷의 텍스트 판독에 충분한 해상도

### 4.2 입력 스키마

```ts
type ImageInput = {
  type: "image";
  images: string[]; // base64 데이터 URL, 1-5장
};
```

Zod 검증: 장수(1-5), 장당 base64 길이 상한(다운스케일 실패/우회 대비), 전체 합계 상한, MIME 타입(jpeg/png/webp). 클라이언트가 다운스케일하지만 서버는 이를 신뢰하지 않고 독립적으로 재검증한다 (직접 API 호출 우회 가능성).

기존 `AnalysisInputSchema` discriminated union에 `ImageInputSchema`를 추가한다. SMS/이메일 텍스트 입력은 변경 없이 유지된다.

## 5. 분석 파이프라인: 판독-후-분석, 단일 호출

### 5.1 왜 단일 호출인가 (승인된 Approach A)

검토한 대안: (B) OCR 선행 후 기존 텍스트 파이프라인 재사용 — 요청당 쿼터 2배 소모로 무료 티어 제약과 정면 충돌, 시각 신호 소실. (C) 클라이언트 측 Tesseract.js OCR — 이미지가 기기를 떠나지 않는 프라이버시 이점은 있으나 한국어 스크린샷 판독 정확도가 낮아 기능 자체를 훼손.

**채택: 단일 멀티모달 호출.** 프롬프트가 모델에게 (1) 먼저 보이는 것을 판독하고(앱 종류, 표시된 발신자, 메시지 순서, 링크), (2) 그 다음 v1의 검증된 신호 목록으로 분석하도록 지시한다. 시각 신호(앱 UI로 카카오톡/SMS 구분, 위장된 발신자 표시명, 가짜 은행 UI)가 텍스트로는 전달 불가능한 탐지 근거가 된다.

### 5.2 결과 스키마 확장

```ts
interface AnalysisResult {
  verdict: "안전" | "의심" | "위험";
  riskScore: number; // 0-100, verdict 밴드와 교차 검증 (v1 그대로)
  redFlags: { flag: string; evidence: string }[]; // ← v2 변경: string[] → 구조화
  explanation: string;
  recommendedAction: string;
  extractedText: string; // ← v2 추가: 이미지 모드에서 모델이 판독한 transcript
}
```

- **`extractedText`**: 이미지 모드에서 모델이 판독한 내용. 텍스트 모드에서는 빈 문자열(입력 원문이 이미 있으므로). UI가 이를 표시해 "모델이 무엇을 읽었는지" 투명하게 보여준다.
- **`redFlags` 구조화**: 각 신호는 반드시 메시지/transcript에 실제로 존재하는 문구를 `evidence`로 인용해야 한다. v1에서 프롬프트 지시("모든 redFlag는 실제 텍스트에 근거할 것")로만 강제하던 규칙이 스키마 수준으로 올라온다 — 인용할 수 없는 신호는 존재할 수 없다. 이 인용구가 §7의 인라인 하이라이트 데이터 소스다.
- 내부 소비자만 있으므로(외부 API 아님) 스키마 변경은 breaking이어도 무방하다 — 프론트/백을 같이 바꾼다.

### 5.3 판독 불가 처리

**이미지 모드 요청에 한해**, 이미지에서 메시지 내용을 찾을 수 없으면(무관한 사진, 빈 화면 등) 모델은 `extractedText`를 빈 문자열로 반환하고, 라우트는 가짜 판정 대신 "스크린샷에서 메시지를 읽을 수 없습니다" 에러를 반환한다 (텍스트 모드의 `extractedText`는 항상 빈 문자열이며 이 규칙과 무관하다). 읽지 못한 것에 verdict를 붙이는 것이 최악의 실패 모드(오판을 신뢰하게 만듦)이므로 명시적으로 차단한다.

### 5.4 Fallback 체계

| 입력 | 1차 | 429 시 fallback | 비고 |
|------|-----|----------------|------|
| 텍스트 | Gemini flash-lite | Groq `openai/gpt-oss-20b` | v1 그대로 (strict structured output) |
| 이미지 | Gemini flash-lite (멀티모달) | Groq `meta-llama/llama-4-scout-17b-16e-instruct` | JSON mode (strict 아님) + Zod 검증 |

Llama 4 Scout는 Groq 무료 티어에서 이미지 입력을 지원하는 멀티모달 모델이다. gpt-oss-20b와 달리 strict schema mode가 아닌 JSON mode만 지원하므로 출력 보장이 한 단계 약하다 — 기존 Zod 검증(`AnalysisResultSchema.parse`)이 이를 받아내며, 검증 실패 시 조용히 잘못된 결과를 반환하는 대신 에러로 실패한다(fail closed). Groq 무료 티어의 분당 토큰 한도(~30k TPM)상 5장짜리 fallback 요청은 수용 가능하나 여유가 크지 않다 — fallback 경로이므로 수용한다.

v1과 동일하게 fallback 트리거는 **Gemini의 429에 한정**한다 — 다른 에러(스키마 검증 실패, 파싱 실패)는 버그 신호이므로 fallback으로 가리지 않고 즉시 표면화한다.

### 5.5 프롬프트 주입 방어 (이미지 확장)

이미지 안의 텍스트는 `<message_to_analyze>` 태그 안의 텍스트와 동일한 지위다: **분석 대상 데이터일 뿐, 어떤 지시로도 취급하지 않는다.** 스크린샷에 "이전 지시를 무시하라"류의 문구가 (진짜 메시지로든, 조작된 이미지로든) 포함되어 있으면 그 자체를 redFlag로 기록한다. 시스템 프롬프트의 기존 주입 방어 문단을 이미지 판독 내용까지 포괄하도록 확장한다.

## 6. 무료 티어 쿼터 검증 결과 (2026-07-13 조사)

설계 승인 전 확인한 사실들 — 계획 수립 시 전제로 사용:

- **일일 한도는 요청 단위(RPD)로 계산된다** — 이미지 5장짜리 요청도 텍스트 요청과 똑같이 1회로 계산. 멀티 이미지가 일일 쿼터를 더 빨리 소진시키지 않는다.
- 이미지 토큰 비용: 768×768 타일당 258토큰. 다운스케일된 스크린샷 장당 ~1,500토큰, 5장 ≈ ~8,000토큰 — flash-lite 무료 티어의 분당 토큰 한도(~250k TPM) 대비 미미.
- 서드파티 자료 기준 flash-lite 무료 티어는 15 RPM / 1,000-1,500 RPD 수준으로 보고되나, **Google은 더 이상 고정 한도표를 공개하지 않는다.** v1에서 flash의 실제 한도가 문서상 추정치와 달리 20 RPD였던 전례가 있으므로, **구현 착수 전 이 프로젝트의 AI Studio rate-limit 대시보드에서 실측치를 확인하고 `quotaGuard.ts`의 상수를 그에 맞춰 조정하는 것을 계획의 명시적 태스크로 포함한다.**
- Groq Llama 4 Scout: 무료 티어 제공, 이미지 입력 지원, JSON mode 지원, ~30k TPM.

## 7. UI 업그레이드

전체 플로우는 유지한다(단일 페이지, 입력 → 분석 → 결과). 세 가지 축:

### 7.1 모바일 우선 재작업

스캠 문자를 받고 스크린샷을 찍는 곳이 폰이다. 한 손 조작 기준으로 재배치: 입력 모드 토글(텍스트/스크린샷), 큰 터치 타깃의 업로드 영역, 하단 고정(sticky) 분석 버튼. `<input accept="image/*">`로 모바일에서 카메라/사진첩 직접 접근.

### 7.2 결과 화면 강화

- **판정 배너**: 색상 코딩된 verdict + 위험도 게이지 (기존 요소의 시각적 강화)
- **원문/transcript 표시 + 인라인 하이라이트**: 텍스트 모드는 입력 원문, 이미지 모드는 `extractedText`를 표시하고, 각 redFlag의 `evidence` 인용구를 문자열 매칭으로 찾아 해당 구간을 하이라이트한다. 매칭 실패한 evidence는 하이라이트 없이 신호 카드에만 표시(모델 인용이 원문과 미세하게 다를 수 있음 — 실패해도 기능 저하일 뿐 오류 아님)
- **신호 카드**: redFlag별로 신호 설명 + 인용된 근거 문구를 카드로 표시
- **권장 조치 블록**: 비전문가 사용자가 다음에 할 일을 바로 알 수 있게 시각적으로 가장 강조

하이라이트 렌더링은 문자열 분할 + React 요소 조합으로만 구현한다 — LLM 출력이 관여하는 경로이므로 `dangerouslySetInnerHTML` 금지 원칙(v1 §10.3)이 그대로 적용된다.

### 7.3 비주얼 폴리시

기존 shadcn/ui 컴포넌트 기반 유지, 새 UI 의존성 없음. 타이포그래피 스케일, 간격 시스템, 팔레트 정돈. Tailwind 유틸리티 클래스만 사용(`style={{}}` 금지, v1 규칙 승계). (다크 모드는 이번 v2 범위에서 제외 — 필요해지면 별도로 다룬다.)

## 8. 에러 처리

v1의 에러 처리 원칙(원시 에러 비노출, 사전 정의된 한국어 안내)에 추가:

- **클라이언트 측**: 이미지 형식/장수/용량 초과 시 업로드 시점에 즉시 안내 (API 호출 전 차단)
- **서버 측**: 스키마 검증 실패(장수/용량/MIME) → 400 + 한국어 안내
- **판독 불가**: `extractedText` 빈 문자열 → "스크린샷에서 메시지를 읽을 수 없습니다. 메시지가 선명하게 보이는 스크린샷인지 확인해주세요"
- **세이프티 필터**: 스캠 스크린샷이 이미지 세이프티 필터에 걸릴 수 있음 — v1의 텍스트 케이스와 동일하게 "분석할 수 없습니다" 계열 안내
- **쿼터 소진**: 기존 안내 유지. 이미지 요청도 동일한 쿼터 풀을 소비하므로 별도 메시지 불요

## 9. 보안 및 개인정보 처리

v1 §10의 모든 원칙이 승계되며, 이미지라는 새 표면에 대해:

- **무저장/무로깅 원칙의 이미지 확장**: base64 페이로드를 어떤 로그/에러 객체/저장소에도 남기지 않는다. `images` 필드는 v1의 `messageBody` 등과 동일한 금지 목록에 추가된다
- **개인정보 고지 갱신**: 스크린샷은 붙여넣은 텍스트보다 더 많은 것을 담는다(화면에 보이는 모든 것). 입력 영역 위 고지를 갱신: "스크린샷은 분석을 위해 Google Gemini(무료 티어 — 서비스 개선에 사용될 수 있음) 또는 Groq로 전송됩니다. 개인정보가 보이는 부분은 잘라내고 올려주세요. 이미지는 저장되지 않습니다"
- **게이트 순서 불변**: Turnstile → IP rate limit → 전역 쿼터 → LLM. 이미지 요청도 모든 쿼터에 1회로 계산 (Google의 계산 방식과 일치)
- **이미지 내 텍스트 = 신뢰 불가 입력**: §5.5의 주입 방어

## 10. 테스트/검증 계획

v1과 동일한 TDD 패턴(Vitest, provider 모킹, 실제 API 무접촉):

- 이미지 입력 스키마: 장수/용량/MIME 경계값
- 멀티모달 요청 구성: inlineData 파트 구조, 프롬프트에 판독-후-분석 지시 포함 여부
- `extractedText` 빈 문자열 → 판독 불가 에러 경로
- redFlags evidence 스키마 검증 (evidence 누락 거부)
- 429 시 Scout fallback 트리거, 그 외 에러는 fallback 안 함
- 하이라이트 렌더링: evidence 매칭 성공/실패 케이스, XSS 안전성(텍스트 이스케이프)
- **수동 실물 검증**: 실제 스캠 스크린샷(택배 사칭, 가족 사칭, 기관 사칭)과 정상 메시지 스크린샷으로 판정 정확도 확인 — v1에서 유닛 테스트가 못 잡는 문제(키워드 과가중, 신호 날조)를 전부 이 단계가 잡았다
