# 안심스캔 — 결과 캐싱(콘텐츠 해시 기반) 설계 문서

- Date: 2026-07-14
- Status: Draft (approved for planning)
- 선행 문서: `2026-07-07-korean-scam-detector-design.md`(v1), `2026-07-13-v2-screenshot-analysis-and-ui-upgrade-design.md`(v2)

## 1. 개요 (Overview)

바이럴로 퍼지는 스미싱/피싱 문자는 다수의 사용자에게 토씨 하나 다르지 않은 동일한 본문으로 전달된다. 현재는 동일한 문자를 100명이 붙여넣으면 100번의 LLM 호출이 발생한다 — 이는 무료 티어 쿼터(Gemini 실측 일일 20회)를 가장 빠르게 소모시키는 낭비이자, 첫 번째 요청 이후로는 사실상 "같은 질문을 다시 하는" 불필요한 지연이다.

**목표**: 동일한 입력(문자/이메일)에 대해서는 최초 1회만 LLM을 호출하고, 이후 동일 입력에는 캐시된 결과를 즉시 반환한다. 결제·유료 모델 전환과 무관하게, 순수 엔지니어링으로 실질적인 처리량과 응답 속도를 개선한다.

**v1 범위**: 스크린샷(이미지) 입력은 캐싱 대상에서 제외한다 — 서로 다른 사용자의 스크린샷은 기기·크롭·압축이 달라 바이트 단위로 일치할 가능성이 사실상 없어, 캐싱 효과가 거의 없다.

## 2. 목표 / 비목표 (Goals / Non-goals)

**목표**:
- 동일한 문자/이메일 입력에 대한 반복 요청을 실제 LLM 호출 없이 즉시 응답
- 기존 아키텍처(Upstash Redis, 무저장 원칙)와 완전히 호환 — 새 인프라나 의존성 추가 없음
- 메시지 본문을 어떤 형태로도 평문 저장하지 않음(v1 §10.3 무기록 원칙 승계)

**비목표**:
- 이미지(스크린샷) 입력 캐싱
- 캐시 수동 무효화/관리 UI
- 캐시 히트 여부를 사용자에게 노출하는 UI(예: "이전에 확인된 메시지입니다" 배지) — 판정 결과 자체는 캐시 히트든 신규 호출이든 동일한 형태이므로 구분해서 보여줄 필요가 없다

## 3. 아키텍처 개요

```
POST /api/analyze
  ├─ 입력 스키마 검증 (기존)
  ├─ Turnstile 검증 (기존)
  ├─ IP rate limit 확인 (기존, 캐시 히트에도 동일하게 적용)
  ├─ [신규] 캐시 조회 — sms/email 입력만 대상
  │    └─ 히트 → 즉시 반환 (전역 쿼터 가드·LLM 호출 모두 건너뜀)
  └─ 미스 (또는 image 입력)
       ├─ 전역 일일/분당 쿼터 가드 확인 (기존)
       ├─ analyzeMessage() 호출 (기존)
       ├─ [신규] 캐시에 결과 저장 (sms/email만, TTL 24시간)
       └─ 결과 반환
```

캐시는 기존에 이미 사용 중인 Upstash Redis에 저장한다 — 새로운 인프라를 추가하지 않는다.

## 4. 캐시 키 설계

`src/lib/analysis/types.ts`의 `AnalysisInput` 중 `sms`/`email` 타입만 캐시 대상이다. 캐시 키는 입력 필드를 타입별로 명시적인 순서로 이어붙인 문자열의 SHA-256 해시로 만든다(범용 `JSON.stringify`는 키 순서가 모호할 수 있어 사용하지 않는다):

```
sms:   sha256(`sms|${senderNumber}|${messageBody}`)
email: sha256(`email|${senderAddress}|${subject}|${body}`)
```

**발신번호/발신 주소를 해시에 포함하는 이유**: 시스템 프롬프트가 발신번호/주소 자체(스푸핑 여부, 도메인 위장 등)를 분석 근거로 사용하므로, 동일한 본문이라도 발신 정보가 다르면 다른 판정이 나올 수 있다. 본문만 해시하면 캐시 히트율은 오르지만 잘못된 판정을 재사용할 위험이 있다 — 정확성을 캐시 히트율보다 우선한다.

## 5. 저장 형식 및 TTL

- Redis 키: `cache:analysis:<sha256hex>`
- 값: `AnalysisResultSchema`에 부합하는 JSON (verdict/riskScore/redFlags/explanation/recommendedAction/extractedText — extractedText는 텍스트 입력이므로 항상 빈 문자열)
- TTL: 24시간. 하루 동안 집중적으로 퍼지는 스미싱 문자를 잡는 데 충분하고, Redis 저장량을 무한정 늘리지 않으며, 시스템 프롬프트를 튜닝한 다음 날부터는 새 판정 로직이 다시 적용된다.

## 6. 에러 처리

- **캐시 조회 실패(Redis 네트워크 오류 등)**: 캐시 미스로 간주하고 정상적으로 새 분석을 진행한다 — 캐시는 최적화일 뿐이므로 실패해도 요청 자체를 막지 않는다(fail open).
- **캐시 저장 실패**: 응답을 반환하기 전에 저장을 `await`하되(기존 코드 스타일과 동일하게 순차 처리), 저장 자체가 실패해도 이를 삼켜서 사용자에게는 정상적으로 분석 결과를 반환한다 — 저장 실패가 이미 성공한 분석 응답을 에러로 바꾸지 않는다(다음 동일 요청에서 다시 캐시를 시도하게 된다).
- 두 경우 모두 에러 객체를 그대로 로그하지 않는다 — 기존 v1 원칙과 동일하게, 예기치 않은 방식으로 메시지 내용이 로그에 섞여 들어가는 것을 방지한다.

## 7. 트레이드오프 (명시적으로 인지하고 감수하는 것)

캐시 키는 메시지 내용의 결정론적 해시다. Redis는 우리 서버 외부에서 조회할 수 없으므로 외부에 정보가 노출되지는 않는다. 다만 내부적으로 "동일한 문구를 이미 누군가 제출한 적이 있으면 응답이 더 빠르다"는 타이밍 차이가 존재한다 — 이는 외부로 노출되는 정보 유출이 아니라 순수 응답 속도의 차이이며, 이 기능이 주는 효율성 이득에 대한 합리적인 대가로 판단해 그대로 받아들인다.

또한 동일 입력에 대해 TTL 기간 동안은 항상 같은 판정이 반환된다(LLM을 매번 다시 호출했다면 미세하게 다른 결과가 나올 수도 있었을 결과가 고정됨) — 이는 부작용이 아니라 오히려 "같은 문자를 붙여넣었는데 사람마다 다른 판정을 받는" 혼란을 줄여주는 바람직한 효과로 본다.

## 8. 테스트 계획

`src/lib/analysis/resultCache.test.ts` (기존 `rateLimit.test.ts`/`quotaGuard.test.ts`의 `@upstash/redis` 모킹 패턴을 그대로 따른다):

- sms 입력에 대한 캐시 히트가 저장된 결과를 그대로 반환하는지
- 캐시 미스 시 `null`을 반환하는지
- 동일한 본문이라도 발신번호/주소가 다르면 다른 캐시 키가 생성되는지
- Redis 조회 중 에러가 발생해도 예외를 던지지 않고 `null`을 반환하는지(fail open)
- `image` 타입 입력은 캐시 조회/저장 대상에서 애초에 제외되는지

`route.test.ts`에는 다음을 추가한다:
- 캐시 히트 시 `analyzeMessage`가 호출되지 않고 캐시된 결과가 그대로 반환되는지
- 캐시 미스 시 기존과 동일하게 `analyzeMessage`가 호출되고, 성공 후 캐시에 저장되는지
- 캐시 히트 시에도 Turnstile 검증과 IP rate limit 체크는 여전히 수행되는지

## 9. 이후 로드맵 (Out of scope, 추후 고려)

- 이미지 입력에 대한 유사도 기반(해시가 아닌) 캐싱 — 완전히 다른 문제 영역(perceptual hashing 등)이라 별도 설계가 필요하다
- 캐시 히트율을 관리자가 볼 수 있는 대시보드
- TTL을 상황에 따라 동적으로 조정하는 정책
